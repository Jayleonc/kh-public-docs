Knowledge Hub 是一套自研的企业知识智能平台。它把文档接入、知识组织、检索、权限治理和 AI 问答放进同一条链路，让企业资料可以被稳定读取、可靠检索，并在明确边界内被模型消费。

如果只看前台，它像一个能读文档、能聊天的系统；继续往里看，会发现它同时覆盖了项目空间、虚拟文件树、文档解析、索引、ACL、RBAC、Git Sync、Worker 和 Agent 工具调用这些层面。

## 一句话概括

Knowledge Hub 是一个面向企业知识场景的 Agent-first 平台，核心目标是让文档在权限受控、来源可追溯、持续可维护的前提下，真正进入问答和业务系统。

这句话里有 4 个关键词，很值得展开：

- `Agent-first`：策略层已经上移，系统能做多步决策
- `企业知识场景`：重点问题集中在权限、更新、治理和可信度
- `来源可追溯`：回答要能回到文档，而不是只停留在生成文本
- `持续可维护`：Git、状态机、Worker 和治理链路都很重要

## 这个项目包含哪些层

### 用户入口层

- `web-chat`：面向公开访问和最终用户问答
- `workspace`：面向内部团队管理知识与协作
- `admin`：面向管理员做权限、配置和治理
- `api`：面向外部业务系统集成

### 核心服务层

- 文档接入与解析
- 项目与虚拟文件树组织
- 向量检索与混合检索
- Agent 工具编排
- ACL / RBAC 权限控制
- Git Sync 与后台 Worker

### 基础设施层

- PostgreSQL + pgvector
- Redis
- 文档解析服务
- LLM Provider

## 这个项目难在什么地方

企业知识系统的难点，通常集中在下面几件事：

- 文档很多，但可信源不明确
- 检索能命中，结论仍然缺少完整上下文
- 页面能藏内容，召回阶段却可能越权
- 文档持续更新，问答结果要跟着一起更新
- 系统需要同时服务公开访问和内部协作

Knowledge Hub 的难度，正在于它没有只解决其中一个点，而是试图把整条链路打通。

## 这个项目的关键特征

这个项目有几个比较关键的特征：

- 访问者可以先读文档，再直接提问，交互感很强
- 每次回答都能暴露系统的设计判断，例如引用、权限边界和全文确认
- 它同时具备产品形态和架构深度，既能讲场景，也能讲实现
- 同一套系统同时承担文档阅读、问答、治理和集成职责

## 适合的使用场景

- 企业产品与 API 文档问答
- 客服知识库和售前资料库
- 内部 SOP、制度、方案和案例查询
- 多租户知识空间管理
- 公开文档站点和受限知识入口

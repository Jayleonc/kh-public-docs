Knowledge Hub 要解决的，是企业知识长期存在、却长期难用的问题。很多团队并不缺文档，缺的是一套能把文档稳定变成“可查、可答、可控、可维护”的系统。

## 如果只看问题表面

表面上看，大家会说：

- 文档太分散
- 用户找不到资料
- AI 回答不稳定
- 权限边界容易出错

这些判断都对，但还不够深入。Knowledge Hub 更关心的是这些问题为什么会反复出现。

## 断点一：文档有了，可信源没有

很多团队同时维护网盘、IM、共享目录、Wiki、仓库和临时 PDF。内容看起来很多，到了问答阶段却会出现几个问题：

- 谁也说不清哪一份才是当前有效版本
- 线上内容和内部原稿常常脱节
- 改完文档以后，问答结果更新时间不确定

Knowledge Hub 倾向让正式资料回到 Git、受控上传或明确的数据源里，再进入统一知识空间。只有源头稳住，后面的问答质量才有基础。

## 断点二：文本进来了，结构丢了

企业文档需要按项目、目录、路径和引用被理解。只有纯文本，没有结构，系统就很难回答下面这些问题：

- 这段话属于哪个产品模块
- 这个 API 在哪个目录下
- 当前答案应当回到哪篇文档继续确认

Knowledge Hub 通过项目、虚拟文件树、文档路径和结构映射保留这些信息，让“文档内容”和“文档位置”同时可用。

## 断点三：检索能召回，权限没跟上

很多系统在页面层能做可见性控制，到了检索层却容易漏掉未授权内容。结果是用户看不到文件，却可能通过问题把片段问出来。

Knowledge Hub 把权限下沉到数据查询和检索链路中：

- 先做租户和身份识别
- 再做项目与资源范围约束
- 检索阶段继续带上 ACL scope
- 目录可见性跟着可访问的内容一起变化

这条链路的目标很直接：无权访问的材料，不进入召回，不进入回答依据。

## 断点四：片段命中了，结论仍然站不稳

企业问答里，片段召回经常只能解决一半问题。用户会问：

- 一个机制为什么这样设计
- 两份文档之间是什么关系
- 这个结论是否有前置条件

单次检索和单段拼接在这里经常不够。Knowledge Hub 采用 Agent + Tools 的问答链路，让系统可以先检索、再精读全文、必要时追问用户，最后输出回答和引用。

## 断点五：上线后缺少治理

知识系统一旦进入真实环境，就会面对持续变更：

- 文档被更新
- 路径被调整
- 权限被修改
- 索引需要重建
- 对外发布内容需要审查

Knowledge Hub 在文档状态、索引状态、异步 Worker、Git Sync 和审计链路上投入了很多设计，目的是让知识系统可以长期维护。

## 这个项目真正想交出的答案

Knowledge Hub 想提供的是一条闭环：

```text
文档源头
  -> 结构化接入
  -> 项目与路径组织
  -> 权限约束
  -> 检索与全文阅读
  -> 带引用回答
  -> 持续同步与治理
```

这条闭环一旦打通，企业知识才能从“堆材料”走向“可被验证地使用”。

这条闭环说明 Knowledge Hub 的重点不在某一个孤立能力，而在整条链路能否一起成立。

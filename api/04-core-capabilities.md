Knowledge Hub 的能力最好按一条链路来看：内容如何进入系统，知识如何被组织，检索如何工作，权限如何约束，最后怎样对外提供服务。这样看，项目的完整性会比一张功能清单更清楚。

## 几个关键能力

Knowledge Hub 当前有 5 个比较关键的能力方向：

- 一套系统同时覆盖文档接入、检索、权限和问答
- Agent 已经进入主决策链路，能做多步工具调用
- 文档路径、目录和结构信息会进入知识组织和问答过程
- 权限约束会继续影响检索与回答依据
- Git Sync 能把文档治理和公开问答串进同一条发布链路

## 1. 内容接入

Knowledge Hub 支持 Markdown、PDF、图片等多种资料进入知识空间。当前系统里，Markdown 很适合做正式文档源，PDF 和图片更适合承接历史材料、截图和复杂版式内容。

这里的关键点不只是“能导入”，还有后续链路是否可控：

- Git 仓库同步，适合长期维护的正式文档
- 手动上传，适合临时材料和补充资料
- 平台内编辑，适合快速迭代的轻量文档
- 文档解析服务，负责处理 OCR、多模态内容和复杂格式

## 2. 知识组织

企业知识如果只有一堆文本，问答质量很难稳定。Knowledge Hub 会保留项目、目录和路径这些结构信息：

- 项目级知识空间，作为内容和权限的第一层边界
- 虚拟文件树，承接目录结构、导航和路径语义
- 文档与 `project_node` 的绑定关系，让组织结构和索引结构保持一致
- 路径与结构映射，用于引用定位、全文阅读和跳转

这样做的价值在于，系统不仅知道“内容是什么”，也知道“内容属于哪里”。

## 3. 检索与阅读

Knowledge Hub 的问答链路不是一次检索拼几段文本就结束。当前主路径会把检索、片段扩展和全文阅读配合起来：

- 向量检索，解决语义匹配
- 关键词检索，解决错误码、版本号、专有名词这类精确命中
- 混合检索，平衡召回范围和命中率
- 上下文扩展，在概念型问题里补足相邻片段
- 全文读取，在片段不足时回到整篇文档确认上下文
- 引用收集，让最终回答可以回溯到原始资料

## 4. 权限治理

Knowledge Hub 把权限当成主链路的一部分来处理：

- 多租户隔离，避免不同租户之间互相影响
- RBAC，控制用户能执行哪些动作
- ACL，控制项目、目录、文档这些资源能被谁访问
- 检索阶段权限过滤，保证无权内容不进入召回结果
- JWT 与 API Key 两种入口凭据，适配用户访问和系统集成

权限进入检索链路以后，知识问答才更适合企业环境。

## 5. 系统集成与交付

Knowledge Hub 同时提供多个对外形态：

- Web Chat，用于阅读和提问
- Workspace，用于内部运营、资料管理和协作
- Admin，用于租户、配置、权限和审计
- API，用于把知识能力接入客服、售前或内部系统

文档发布方面，Git Sync 很关键。它让文档修改、审查、同步和索引形成一条清晰的发布链路。

## 6. 运行时能力

Knowledge Hub 在运行时还承担几类容易被忽略、实际很重要的能力：

- 文档状态和索引状态分离，便于追踪解析与重建索引
- Worker 异步处理，降低上传与索引之间的耦合
- 路径同步、孤儿资源清理、摘要更新等后台治理任务
- 流式事件和引用回传，便于前端消费 Agent 的过程事件与结果

这些能力共同决定了系统能否从一个“能跑的 Demo”走向“能长期维护的平台”。

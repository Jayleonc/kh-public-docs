Knowledge Hub 当前的知识问答主路径已经进入 Agentic RAG 形态。换句话说，系统会先由 Agent 判断该做什么，再把检索、目录浏览、全文读取和追问用户这些动作当成工具来调用。

## 为什么要走到 Agent-first

固定检索流水线在简单问题里很高效，例如：

1. 收到问题
2. 执行检索
3. 拼接上下文
4. 生成回答

这条路径面对复杂问题时会暴露几个限制：

- 问题可能包含多个意图
- 片段命中了，证据仍然不够完整
- 用户给出的路径、截图或历史上下文需要先被理解
- 问题本身不够清楚，需要先追问

Knowledge Hub 当前把 Agent 放在策略层，就是为了处理这些更复杂的决策。

## 当前运行时长什么样

源码里的 Agent 运行时采用 `OpenAIFunctionsAgent + Executor`，最大迭代次数是 `5`，执行超时控制在 `90s`。当前主执行链挂载了 4 个工具：

- `search_knowledge_base`：进入检索层，负责从知识库拿证据
- `read_document`：回到整篇文档，防止只看片段就下结论
- `get_folder_info`：查看目录结构，补足路径和导航信息
- `ask_user_clarification`：当用户意图不清时，主动要求补充上下文

代码库里还有候选工具文件，但当前主干刻意只挂载这 4 个。这个取舍很重要，它减少了工具选择面，提升了行为稳定性。

## Agent 如何被约束

Knowledge Hub 没把 Agent 交给“自由发挥”。当前提示词里有一套明确的工作流约束：

### 阶段零：Memory Scan

Agent 先看会话历史。如果历史里已经有足够的文档片段、全文内容或结论，就直接基于现有上下文回答，不重复检索。

### 阶段一：Scout

只有历史不足时，才允许调用 `search_knowledge_base`。如果第一次没有命中，系统要求继续改写查询或缩减关键词，而不是立刻放弃。

### 阶段二：Deep Read

检索返回的是片段，不是完整上下文。一旦关键信息被截断，或者结论站不稳，Agent 必须继续调用 `read_document` 去读全文，再决定最终答案。

这套约束的意义在于把 Agent 的自由度压进一个更可靠的框里，让多轮推理服务于证据，而不是脱离证据。

## RAG 在这里仍然很重要

Agent 上移以后，RAG 的价值没有下降，只是职责更清晰了。RAG 继续负责：

- 向量检索和关键词检索
- 上下文扩展
- 路径过滤
- ACL scope 下的受限召回
- 引用片段组织

Knowledge Hub 当前还有一个检索内部使用的意图探测器，用来区分 `code / howto / concept / factoid` 这类问题，再决定是否扩展兄弟片段。这种判断服务于检索质量，不承担 Agent 的整体决策职责。

## 追问机制为什么值得单独写

`ask_user_clarification` 在源码里不是普通文本回复。它会触发一个专门的澄清事件，并用哨兵错误中断当前执行循环，把控制权还给用户。

这个设计很实用，因为它把“需要补充信息”建模成一次受控退出。系统不会假装自己已经理解用户，也不会带着模糊上下文硬往下跑。

## 多模态输入怎么接进来

当用户输入里带图片时，Knowledge Hub 会先经过一个视觉摘要阶段，把图片整理成结构化观察，再交给主 Agent 继续推理和检索。这让系统可以处理截图、界面说明和图文混合问题。

## 这套设计的代价

Agent-first 的收益很明显，代价也同样真实：

- 延迟会高于固定流水线
- Prompt 和工具边界必须设计得更严
- 可观测事件流和引用链路必须跟上
- 工具面一旦膨胀，行为稳定性会下降

Knowledge Hub 当前接受这些代价，因为它更看重复杂问题处理能力、全文确认能力和公开场景下的可核验性。

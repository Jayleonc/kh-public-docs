Knowledge Hub 当前的查询入口不是简单的固定 RAG Pipeline，而是由 Agent 驱动的知识问答流程。RAG 在这里是工具能力，Agent 负责判断什么时候检索、什么时候读取全文、什么时候要求用户澄清。

## 固定 RAG 的限制

传统 RAG 通常是：

1. 接收问题
2. 向量检索
3. 拼接上下文
4. 生成回答

这个流程适合直接问题，但遇到复杂问题时容易不足。例如用户的问题可能包含多个意图，检索结果可能只命中片段，或者需要对比多份文档才能回答。

## Agentic RAG 的做法

Agent 可以使用一组受控工具：

- 搜索知识库
- 读取文档全文
- 查看目录信息
- 请求用户澄清
- 根据上下文生成最终回答

这样系统可以先检索，再判断是否需要继续读取文档，最后给出带引用的回答。

## 为什么保留 RAG

Agent 不替代 RAG。Agent 负责决策，RAG 负责高质量检索。两者分工不同：

- RAG 解决“从哪里找资料”
- Agent 解决“下一步该做什么”
- 引用系统解决“答案依据是什么”
- 权限系统解决“哪些资料可以被使用”

## 回答质量控制

对外文档场景下，回答应该遵守三条规则：

1. 优先基于公开文档回答
2. 找不到依据时明确说明没有找到
3. 对关键结论提供引用来源

这比“模型知道什么”更重要，因为客户需要的是可信的产品说明，而不是开放域聊天。

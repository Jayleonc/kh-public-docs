Knowledge Hub 的安全设计建立在一个很朴素的目标上：用户看到什么、搜到什么、能拿什么做回答依据，这三件事要保持一致。

## 安全边界分成哪几层

### 租户边界

每个租户拥有自己的项目、文档、配置和策略。请求进入系统后，先经过认证与租户识别，再进入后续业务逻辑。

### 项目边界

项目是知识空间的管理边界。它把文档、目录结构、权限和问答范围组织到一起，方便公开访问和内部协作分别治理。

### 资源边界

Knowledge Hub 支持把访问策略挂到项目、目录和文档这些资源上。当前 ACL 里可以表达：

- public
- private
- partial

`partial` 还能继续细分到用户、角色、用户组和部门。

## RBAC 和 ACL 如何分工

RBAC 解决“允许做什么动作”，例如查看项目、上传文档、修改权限、管理租户配置。

ACL 解决“允许看到什么资源”，例如某个目录是否可见、某篇文档是否可读、某次检索是否可以命中这份内容。

这个分工很重要。只有把动作权限和资源权限拆开，系统才能在多租户和多角色环境下保持清晰。

## 为什么 deny-by-default 很关键

当前 ACL 服务在几个场景下会直接拒绝访问：

- 关键上下文缺失
- 策略缺失
- 策略解析失败

这种 deny-by-default 策略会让默认体验更保守，但它符合企业知识场景的安全预期。系统宁可少放一些内容，也不应因为上下文缺失而意外放开。

## 权限为什么要进入检索层

Knowledge Hub 的一个核心判断是：知识问答的真正风险通常不在页面可见性，而在召回阶段。

所以当前实现会把 `KnowledgeSpace` 转成查询 scope，并把 ACL 条件带进数据库查询和检索范围构造中。对于虚拟文件树，目录是否可见也会跟着子节点的可访问性一起变化。

这意味着：

- 无权文档不进入召回结果
- 目录可见性与内容权限保持一致
- AI 回答使用的证据集天然受限

## 公开访问环境的推荐治理方式

如果要把 Knowledge Hub 作为公开站点或 Demo，推荐这样隔离：

- 使用独立租户或独立公开项目
- 公开资料和内部资料彻底分仓
- 外部访问只给只读和问答能力
- 开启限流、审计和最小权限凭据
- Git 仓库审查通过后再同步到公开项目

这套做法的目标很直接：让公开访问环境只承载公开资料，让系统边界和运营边界一起清楚。

## 这套设计的代价

更严格的权限模型会带来几类成本：

- 查询和检索逻辑更复杂
- 目录可见性规则需要更仔细处理
- 策略继承、部门和用户组解析需要更多治理工作

这些复杂度是值得接受的，因为企业知识系统一旦越权，问题通常比“答慢一点”严重得多。

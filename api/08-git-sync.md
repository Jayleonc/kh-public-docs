Knowledge Hub 很适合把正式文档放进 Git，再通过同步进入知识空间。这样做的收益不只在版本管理，更在于它能把文档治理和知识问答绑成同一条发布链路。

## 为什么 Git 会成为文档源头

正式对外文档通常需要下面这些能力：

- 清楚知道谁改了什么
- 审查敏感信息和措辞风险
- 出现问题时可以快速回滚
- 让线上内容和仓库内容保持一致

Git 天然擅长处理这些问题，所以很适合成为公开文档和客户文档的可信源。

## 同步链路是什么样

对外文档进入 Knowledge Hub 的典型路径如下：

```text
Git 仓库
  -> Pull Request 审查
  -> 合并到发布分支
  -> Git Sync
  -> 更新项目节点与文档内容
  -> 触发后续索引
  -> Web Chat 对外提供阅读与问答
```

这条链路一旦建立起来，文档更新、知识更新和对外回答质量会一起前进。

## 为什么这比长期手动上传更稳

手动上传适合临时材料，长期作为正式发布路径会遇到几个常见问题：

- 仓库里的版本和线上版本脱节
- 修改缺少代码审查和回滚记录
- 同一份材料出现多处副本，越来越难对齐
- 文档更新和问答更新之间的责任边界不清楚

Git Sync 把这些问题收束回一个主线：仓库更新驱动公开内容更新。

## Knowledge Hub 在这里做了哪些配合

Git Sync 在系统里不只是“拉文件”。它还会影响：

- 文档与项目节点的绑定关系
- 路径和导航结构
- 后续索引与问答数据
- 公共内容与维护内容的可见性

当前公开文档仓库采用 `api/` 作为可见内容目录，也正是为了让同步钩子能稳定地识别什么对外可见、什么只给维护者看。

## 为什么 Git 绑定项目要收紧上传入口

一旦项目把 Git 仓库当成正式源头，系统就需要尽量避免旁路写入。否则很容易出现：

- 仓库改了一版
- 平台里又手动传了一版
- 用户看到的是第三种状态

Knowledge Hub 在跨模块协调层对这件事有明确约束，目的就是守住“正式内容从哪里来”这条边界。

## 推荐发布流程

1. 在 Git 仓库修改 Markdown 文档
2. 发起 Pull Request
3. 审查表述、结构、敏感信息和公开边界
4. 合并到发布分支
5. 触发 Git Sync
6. 检查导航、引用和问答效果

## 什么时候仍然适合手动上传

手动上传依然有价值，尤其适合这些场景：

- 临时演示材料
- 客户临时提供的一次性附件
- 还没进入正式文档流程的草稿
- 需要快速验证解析效果的测试文件

正式对外介绍文档更适合放回 Git，这样整套系统会更容易长期维护。

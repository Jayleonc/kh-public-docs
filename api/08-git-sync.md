对外客户文档应该优先使用 Git Sync，而不是长期依赖手动上传。

手动上传适合临时材料，但正式文档需要可审查、可回滚、可追踪。Git 正好提供这些能力。

## Git Sync 的价值

**版本清晰**

每一次文档修改都有 commit 记录，能知道谁改了什么、什么时候改的、为什么改。

**审查明确**

对外文档可以通过 Pull Request 审查，避免把内部路径、客户名称、密钥、未公开能力或不准确表述同步出去。

**回滚简单**

如果发布后发现问题，可以回滚 commit，再触发同步。

**避免内容漂移**

手动上传容易造成“本地一版、线上一版、知识库里又一版”。Git Sync 可以让 Git 仓库成为唯一可信源。

## 推荐发布流程

1. 在 Git 仓库修改 Markdown
2. 提交 Pull Request
3. 审查内容和敏感信息
4. 合并到发布分支
5. Knowledge Hub 执行 Git Sync
6. 检查文档导航和问答质量

## 当前 web-chat 渲染约定

当前文档仓库按 Knowledge Hub 个人版规则组织：

- 对外可见 Markdown 放在 `api/` 下
- `api/_meta.yaml` 定义目录显示名和排序
- 每个 Markdown 文件的 front matter 第一项是 `title`
- 文件名使用数字前缀控制排序
- 根目录不放 `_meta.yaml`

## 手动上传何时使用

手动上传可以作为补充方式，适合：

- 临时演示材料
- 一次性客户文件
- 非长期维护的 PDF
- 还没进入正式文档流程的草稿

正式对外介绍文档不建议用手动上传作为主路径。

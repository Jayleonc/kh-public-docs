Knowledge Hub 当前可以通过 Web Chat、Workspace、Admin 和 API 这几种形态接入。它们服务的角色不同，暴露的能力边界也不同。

## Web Chat

Web Chat 最适合公开访问和最终用户访问。它承担的事情很聚焦：

- 阅读公开或授权文档
- 直接提问
- 查看引用和路径信息
- 在低学习成本下体验系统能力

如果你的目标是客户自助查询、产品文档问答或公开文档入口，Web Chat 通常是第一选择。

## Workspace

Workspace 更适合内部业务团队和运营团队。它承接的是知识生产和维护流程，例如：

- 创建知识空间
- 上传资料
- 选择参与问答的内容范围
- 管理协作和内部工作流

这类能力更偏内部运营，不适合直接暴露给公共访问者。

## Admin

Admin 面向技术管理员，负责：

- 租户配置
- 权限与角色管理
- 系统治理
- 审计和运维相关设置

它承担的是控制面能力，通常应与对外入口分离。

## API

API 用于把 Knowledge Hub 的知识能力嵌入其他系统，例如：

- 客服机器人
- 销售和售前助手
- 内部运营平台
- 客户门户
- 自动化知识流程

API 接入时应当绑定明确的租户、项目、权限和限流策略，避免让下游系统拿到过大的能力面。

## 公开访问时的推荐架构

公开访问场景建议采用这条链路：

```text
客户浏览器
  -> Web Chat
  -> 受限凭据或代理层
  -> Knowledge Hub 公开项目
  -> 公开文档集合
```

这条路径的重点是“入口轻、权限紧、内容清楚”。它能让外部访问者体验问答和引用，同时把管理面完全留在内部。

## 内部使用时的推荐分工

- Web Chat：面向阅读和问答
- Workspace：面向知识维护和协作
- Admin：面向权限与治理
- API：面向系统对系统集成

把角色和入口拆开以后，系统边界会更稳，权限也更容易收敛。

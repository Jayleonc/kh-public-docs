## Knowledge Hub 可以如何概括？

Knowledge Hub 是一套自研企业知识智能平台。它把文档接入、项目空间、虚拟文件树、检索、权限治理和 Agent 问答放进同一条链路，让企业资料可以被稳定查询、带引用回答，并在明确权限边界内持续维护。

## 这个项目的代码是怎么写出来的？

Knowledge Hub 的前后端实现高度依赖 AI 工具生成。作者投入的主要工作，是定义问题、决定架构边界、设计约束体系、拆分执行阶段、验收生成结果和持续复盘。

## 为什么还要把 AI 开发方式单独写出来？

因为这套约束和工作流本身就是项目事实的一部分。代码如何生成、如何被约束、如何被验收，会直接影响系统如何演进，也会影响这套项目文档应该如何理解。

## 这个项目目前有哪些实际使用方式？

当前已经包括智能客服知识基座、SaaS 与运营团队的知识管理、面向客户的内容入口，以及 AI 应用部门通过 API 管理 prompt 相关能力。

## Knowledge Hub 更接近知识库还是 Chatbot？

它更接近一套企业知识系统。聊天只是用户入口，系统背后还包含文档接入、虚拟文件树、检索、权限、审计和 Agent 调度。

## 为什么当前要强调 Agent-first？

因为企业问题经常需要多步动作：先检索，再看目录，再读全文，必要时还要追问用户。把 Agent 放到策略层，可以把这些动作组合起来。

## 这套系统是一次设计完成的吗？

不是。Knowledge Hub 经历了从基础检索、到更复杂的 RAG 编排、再到当前 Agent-first 形态的演进。这个过程里既有能力新增，也有大量主动删减和简化。

## 既然用了 Agent，为什么还保留 RAG？

因为 Agent 负责决策，RAG 负责取证。没有可靠检索，Agent 就缺少可信上下文；只有检索，没有策略层，又很难处理复杂问题。

## 为什么要把引用看得这么重？

企业知识场景里，用户通常需要回到原文确认上下文。引用不仅提升可信度，也能帮助排查文档缺口和回答边界。

## 为什么选择 PostgreSQL + pgvector？

这个选型可以把业务数据和向量数据放在同一套基础设施里，运维面更收敛，适合当前阶段的系统复杂度。代价是大规模检索优化空间会比专用向量引擎小一些。

## 为什么要单独说明技术取舍？

因为 Knowledge Hub 的价值很大一部分来自工程判断。是否采用 Agent-first、是否把权限压进检索层、是否让 Git 成为可信源，这些选择都会直接影响系统的稳定性和可解释性。

## Git Sync 的核心价值是什么？

核心价值在于让仓库成为文档可信源。这样文档修改、审查、回滚、同步和问答更新可以落到同一条链路上。

## 权限具体会影响哪些地方？

权限会影响页面可见性、目录可见性、资源访问范围和检索结果。Knowledge Hub 的目标是让“能看到的内容”和“能拿来回答的内容”保持一致。

## 文档更新后，多久能进入问答？

文档需要先完成同步，再经过解析和索引。索引完成后，新内容才会稳定进入问答链路。Knowledge Hub 之所以维护独立的索引状态，就是为了把这个过程追踪清楚。

## 如果文档里没有答案，系统会怎么做？

理想行为是明确说明当前公开资料没有足够证据，而不是补充没有来源的结论。Knowledge Hub 当前的提示约束也在推动这种回答风格。

## 客户能直接使用 Workspace 或 Admin 吗？

通常不建议。公共访问更适合放在 Web Chat，Workspace 和 Admin 更适合内部团队和管理员使用，因为它们包含更高权限的能力。

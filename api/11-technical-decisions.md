这一页把 Knowledge Hub 几个关键技术决策单独拎出来，说明背后的约束、收益和代价。

## 1. 为什么选择 Agent-first

Knowledge Hub 当前把 Agent 放在策略层，把 RAG 放进工具层。这个决策来自真实问题，而不是概念追新。

固定 RAG 流水线在简单问答里效率很高，但企业知识问答经常会遇到这些情况：

- 一个问题包含多个意图
- 一次召回拿到的只是局部片段
- 用户给了截图、路径或上下文历史
- 当前信息不够，需要先澄清

Agent-first 能把这些动作组合起来：先看历史，再检索，再精读全文，必要时继续追问。

收益：

- 复杂问题的处理能力更强
- 工具可以按需要组合，而不是绑定死流程
- 更适合多模态输入和多轮对话

代价：

- 延迟和推理成本更高
- Prompt、工具边界和可观测性要求更高
- 如果工具面失控，行为稳定性会明显下降

Knowledge Hub 当前接受这个代价，因为它服务的是复杂知识场景，不是极短路径 FAQ。

## 2. 为什么使用 PostgreSQL + pgvector

Knowledge Hub 当前没有引入独立向量数据库，而是把向量检索放进 PostgreSQL + pgvector。

这个决策的考虑很直接：

- 主业务数据本来就在 PostgreSQL
- 向量数据和业务数据放在一起，事务语义更自然
- 运维面更收敛，部署复杂度更低
- 对当前数据规模和团队阶段更友好

收益：

- 少一套基础设施
- 数据模型、权限查询和检索链路更容易协同
- 对中小规模知识库足够实用

代价：

- 极大规模向量检索下，调优空间不如专用引擎
- 一些高阶索引能力和检索特性需要自己权衡

这属于典型的阶段性选型。Knowledge Hub 更看重整体系统复杂度的可控，而不是过早把检索层做成单独平台。

## 3. 为什么强调 Project 第一性

Knowledge Hub 的知识消费方式，更接近“知识空间”而不是“向量集合”。项目这一层存在，系统才更容易把这些问题说清楚：

- 这份知识属于哪个业务空间
- 当前对话应该落在哪个上下文里
- 哪些目录、文档和权限策略是一组

源码层面也已经在往这个方向收敛，例如会话绑定 `project_id`，而不是只绑定数据集。

收益：

- 知识组织和用户心智更一致
- 权限、路径和问答范围更容易统一
- 更适合多租户和协作场景

代价：

- 旧模型向新模型迁移时，需要处理兼容关系
- 模块边界会比“只围绕数据集”更复杂

## 4. 为什么文档状态和索引状态要拆开

Knowledge Hub 把文档处理状态和索引一致性状态分开维护，这是一个很有工程价值的决定。

文档处理状态回答的是：

- 文档是否上传成功
- 是否完成解析
- 是否完成向量化

索引一致性状态回答的是：

- 当前索引是否仍然与内容一致
- 内容改了以后是否需要重建
- 权限变化后是否需要刷新可检索范围

收益：

- 运维和排障时更容易定位问题
- 可以支持按需重建索引，而不是每次都从头处理
- 内容变更和治理动作可以更精确地影响后续流程

代价：

- 状态管理更复杂
- Worker、接口和前端都要理解这两套语义

## 5. 为什么把权限下沉到检索阶段

企业知识系统的核心风险之一，是无权内容被检索结果间接暴露。Knowledge Hub 当前把权限前置到检索范围构造和资源查询中，这个决策非常关键。

收益：

- 无权内容在召回阶段就被排除
- AI 使用的证据集天然受限
- 目录导航和内容可见性更容易保持一致

代价：

- 查询条件更复杂
- ACL、组织结构和 VFS 之间要做更多协同
- 性能调优会更细致

这类复杂度很难省掉，因为越权检索的后果通常比“稍慢一点”严重得多。

## 6. 为什么正式文档优先走 Git Sync

Knowledge Hub 希望把正式资料放进一条可审查、可回滚、可追责的发布链路里。Git Sync 正好把这些能力串起来。

收益：

- Git 仓库成为唯一可信源
- 文档更新与公开问答更新保持一致
- 审查和敏感信息治理更容易执行

代价：

- 发布路径会比直接上传更重
- 内容修订要接受代码审查节奏

对于公开站点和长期客户文档，这个代价通常是值得的。

## 7. 为什么工具集保持克制

当前主执行链只挂了 4 个工具，这体现的是刻意收敛，不是能力面缺失。

工具越多，Agent 的动作空间越大，调试和治理成本也越高。Knowledge Hub 当前更强调：

- 先把常用主链路打稳
- 保证每个工具职责足够清晰
- 让事件流和引用链路更容易解释

这是一个典型的工程判断：先做稳定系统，再逐步扩能力面。

Knowledge Hub 的很多设计都指向同一个目标：让知识系统更可信、更可维护、更适合真实企业环境。它愿意接受一定的复杂度和成本，换取更稳的边界和更强的解释力。

Knowledge Hub 的当前形态，与它的演进路径密切相关。把这条演进线讲清楚，更容易理解系统为什么会变成今天这样。

## 阶段一：基础检索能力

最早期的目标很直接：把文档接进来，让用户能搜到相关内容，再基于检索结果生成回答。

这个阶段解决的是：

- 文档怎样进入系统
- 向量检索怎样跑通
- 回答怎样最基本地回到文档

这是知识系统必须先迈过去的一步，但它很快会遇到上限。复杂问题、路径语义、全文确认和权限边界都会持续逼近。

## 阶段二：Advanced RAG / IDRA

下一阶段开始把检索做细。系统引入了更多检索编排能力，例如：

- 查询意图分类
- 路径语义
- 混合检索
- 上下文扩展
- 多阶段检索策略

这一步的价值在于，它修复了“一套检索参数打天下”的问题，让系统对不同问题类型有更细的处理。

但这一步也暴露了一个新问题：底层编排越来越复杂，策略越来越多，很多判断在运行时其实更适合交给上层。

## 阶段三：Agent 进入主链路

当知识问答开始面对多轮问题、目录浏览、全文阅读和澄清需求时，Agent 的价值开始明显上升。

系统开始具备这些能力：

- 多轮工具调用
- 根据上下文决定是否继续检索
- 必要时读全文，而不是只看片段
- 用户意图不清时主动追问

这个阶段说明一个事实：复杂知识问答已经不再只是“检索做得更准”，还需要策略层。

## 阶段四：Agent-first / Agentic RAG

当前版本进一步把决策权上移。Agent 成为知识问答主入口，RAG 收敛为工具能力，系统也开始具备多模态协作、记忆扫描和更明确的护栏约束。

这一阶段的关键变化包括：

- `search_knowledge_base` 成为检索入口工具
- `read_document` 让全文确认进入主路径
- `ask_user_clarification` 让澄清成为受控动作
- `get_folder_info` 让目录语义也能进入问答过程

从这里开始，Knowledge Hub 更像一套策略驱动的知识系统，而不只是一个检索流水线。

## 演进过程中删掉了什么

Knowledge Hub 的演进并不只是“不断加能力”，还有大量主动删减。原始重构记录里，删掉了不少模块和内部编排逻辑，例如：

- 检索预设系统
- 独立 Planner
- 文档质量检查模块
- AI 文档重构模块
- 部分多阶段检索编排逻辑

这些删除动作背后有几个共同原因：

- 实际用户不会正确配置复杂参数
- 某些模块提供的是回答形式，不是底层新能力
- 上层已经具备重试和决策能力，底层无需重复做一遍
- 代码复杂度与实际收益不匹配

这类删减同样重要，因为它体现了架构如何收敛。

## 几条很关键的经验

### 1. 运行时决策比静态配置更重要

当 LLM 已经能理解具体 query，并在运行时做出合理判断时，过重的配置系统很容易变成负担。

### 2. 上层有重试能力，底层就要收敛职责

如果 Agent 已经能在工具层完成“搜不到就换策略重试”，底层检索层继续内建复杂重试和多阶段循环，收益会快速下降。

### 3. 代码删除本身也是架构进展

删掉上万行代码却没有削弱核心能力，往往说明系统边界更清楚了。Knowledge Hub 的重构记录里，这一点非常明显。

### 4. 企业知识系统的难点在治理

越往后做，越会发现决定系统质量的关键不只在检索准确率，还在：

- 权限边界是否可靠
- 文档更新是否可追踪
- 索引状态是否可观察
- 发布链路是否受控

## 这条演进线说明了什么

它至少说明了两件事：

- 系统是被真实问题一步步推着演进出来的，不是概念拼装的产物
- 每一次演进都带着明确的约束、代价和取舍

比起只陈述“支持 Agentic RAG”，这条演进线更能解释系统为什么会采用当前结构。

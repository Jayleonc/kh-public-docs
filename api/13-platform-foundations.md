---

如果只从第一性原理看，企业知识平台能不能经得住考验，核心通常不在花哨功能，而在底盘模块是否扎实。Knowledge Hub 当前的架构重点，也集中在这些基础模块上。

## 1. 内容进入与索引链路

这是平台的“进食系统”。文档接入、解析、切分、元数据提取、嵌入生成、索引写入，只要任何一个环节不稳定，后面的检索和问答都会被拖垮。

Knowledge Hub 当前把这条链路拆成了比较明确的阶段：

- 接收文档
- 触发解析
- 生成切片
- 生成向量
- 写入索引
- 更新文档与路径状态

这种拆法的价值，在于每一段都能独立观察、重试和恢复。

## 2. 元数据合约

很多系统把重点放在“能搜到”，却忽略“搜到以后能不能解释、过滤和回跳”。Knowledge Hub 当前对片段结果保留了 `project_id`、`source_type`、`ref_id`、`project_node_id`、`vfs_path`、`header_path`、`structure_path` 这些结构信息，原因就在这里。

这类字段决定了很多后续能力能不能成立：

- 项目内搜索
- 权限过滤
- 路径限定
- 结果回跳
- 文档定位
- 审计溯源

元数据合约一旦缺位，平台很容易不断返工。

## 3. 权限与多租户隔离

企业平台和玩具系统的分水岭，通常就在这里。Knowledge Hub 当前把权限做成了多层约束：

- 租户隔离
- RBAC 动作权限
- ACL 资源权限
- 检索阶段实时过滤

最关键的一点，是权限不能只停留在页面层。只有把权限下沉到检索阶段，无权内容才不会进入召回结果和回答依据。

## 4. Project / VFS 内容组织层

没有内容组织层，知识系统很容易退化成一堆孤立片段。Knowledge Hub 当前把 `Project` 作为逻辑知识单元，把 `VFS` 作为路径与层级承载层，这使得系统可以同时保留：

- 项目边界
- 目录关系
- 文档路径
- 节点定位
- 权限继承

当 Git Sync、项目内搜索、文档导航和权限继承都要同时成立时，这一层会变得非常关键。

## 5. 状态机与增量更新

知识库不是一次性构建完成的静态系统。它会持续面对：

- 上传
- 同步
- 失败重试
- 重复触发
- 内容修改
- 路径变更
- 权限调整

Knowledge Hub 当前把文档状态和索引状态拆成两套状态，目的就是处理这些现实问题。没有状态机，异步链路经常会走向偶发性故障和难以解释的不一致。

## 6. 搜索与导航闭环

用户的目标通常是“找到正确材料并回到原文位置”，而不是只获得一段 embedding 相似的文本。Knowledge Hub 当前在检索结果里保留 `project_node_id`、`vfs_path`、`structure_path`，就是为了让搜索结果可以重新接回：

- 项目目录
- 文档节点
- 章节位置
- 原始上下文

搜到了却不能回去确认，平台体验仍然会很差。

## 7. 可观测性与审计

平台越接近企业环境，越不能只靠日志猜问题。Knowledge Hub 当前已经把监控、日志、审计和治理链路放进架构边界里，原因很直接：

- 需要知道索引是否成功
- 需要知道检索和问答延迟
- 需要追踪失败重试和队列积压
- 需要记录谁访问了什么内容
- 需要识别模型、索引和内容版本之间的关系

这类能力平时看起来不显眼，出问题时会成为最关键的保命装置。

## 如果要做优先级排序

从平台可靠性的角度看，Knowledge Hub 这类系统里更靠前的顺序通常是：

1. 权限隔离
2. 索引链路
3. 元数据合约
4. Project / VFS
5. 状态机与可观测性

前两项决定平台能不能安全上线，中间两项决定平台能不能长期演进，最后一项决定出了问题时能不能迅速定位。

## 这一页的核心判断

企业知识平台最重要的事情，始终是知识有没有被正确摄入、正确组织、正确过滤、正确找回、正确解释。少掉任何一个“正确”，系统都会在压力上来之后暴露问题。

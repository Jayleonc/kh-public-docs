Knowledge Hub 的一个特殊之处，在于它的前后端主要实现都由 AI 工具生成。作者没有沿用传统方式逐行手写前后端代码，主要精力放在问题定义、架构判断、约束设计、阶段拆分、结果验收和长期迭代上。

理解这件事很重要，因为它直接影响这个项目应当怎样被阅读。代码只是最终产物的一层；前面的判断、约束和后面的复盘，决定了系统能否长期成立。

## 1. 代码是怎样产生的

从实现方式看，Knowledge Hub 的前后端实现高度依赖 AI 工具产出。实际使用过的工具包括：

- Claude Code
- Codex
- Antigravity

这些工具承担了代码生成、局部重构、方案对比、文档整理和问题排查等工作。每次生成都要落在明确的目标、边界和验收条件里。

## 2. 作者主要负责什么

作者在这个项目里的主要工作，集中在下面这些事情：

- 定义问题和目标
- 决定架构边界
- 识别真正的难点
- 设计约束文件和工作流
- 拆分实现阶段
- 审查 AI 生成结果
- 判断哪些模块应该保留，哪些模块应该删除
- 把演进事实回写进文档

如果把工程产出拆开看，代码只是其中一层。更上游的部分，是如何把一个模糊想法变成可执行的问题；更下游的部分，是如何确认结果可信、结构稳定、后续还能继续演进。

## 3. 这套开发方式如何被组织起来

Knowledge Hub 的开发过程，并不是一轮长 prompt。当前做法更接近一套结构化流程：先澄清意图，再编译约束，再分阶段执行，最后独立复审和回写。

这套方法的核心有两部分：

- SDD：先定义意图和约束，再进入实现
- skill 系统：把不同类型的任务交给不同工作流处理

在当前相关工作流材料里，这条主链会被拆成几段：

- `idea / quick / bug`：入口分流
- `spec / plan / task`：把意图编译成可执行结构
- `execute / review-exec / sync-doc`：执行、复审和文档回写

这些名字本身不是重点。真正重要的是阶段边界被固定下来，AI 在不同阶段承担的职责也会被固定下来。

## 4. 约束文件和 skill 系统起什么作用

当前仓库里，可以直接看到几类约束材料：

- `CLAUDE.md`
- `constitution.md`
- `.claude/skills/*`
- `.codex/skills/*`

`CLAUDE.md` 说明了仓库性质、外部源码路径和基本工作方式。`constitution.md` 负责更硬的事实约束，例如引用代码前必须先读源码，不能靠猜测补实现。`skills` 则把常见任务整理成更稳定的工作流入口，例如 `grow`、`refactor`、`study`、`pdf`、`greet` 等。

这些文件把个人经验外化成可重复执行的约束。模型每次进入任务时，都能在同一套边界里工作，漂移会更小，复盘也更容易。

## 5. 为什么这里会强调思考大于代码

这个项目里，很多关键决策发生在写代码之前和写代码之后。

写代码之前，要先判断很多事情：

- 问题要不要做
- 系统边界画在哪里
- RAG 检索协议需要哪些元数据
- 权限应该落在页面层、接口层还是检索层
- Project / VFS 是否应该成为组织主轴

写代码之后，还要继续判断：

- 生成结果是否真的符合目标
- 哪些模块带来了额外复杂度
- 哪些能力应该被删除，哪些能力值得继续投入
- 文档、代码和实际行为是否仍然一致

所以，这个项目里真正稀缺的部分，是长期判断。代码生成速度本身并不稀缺，判断失误带来的返工成本反而更大。

## 6. 这种方式带来了什么变化

这种开发方式带来了几类明显变化：

- 原型和重构推进更快
- 架构思路更容易快速验证
- 文档、代码和讨论可以同步推进
- 很多重复劳动可以交给 AI 处理

同时，它也提出了新的要求：

- 约束必须更明确
- 验收必须更严格
- 事实校验必须前置
- 复盘和文档回写不能缺位

如果这些配套条件跟不上，生成速度只会把问题更快放大。

## 7. 这篇文档想说明什么

Knowledge Hub 既是一套企业知识平台，也是一份 AI 辅助开发实践的真实样本。这里值得看的内容，包括代码如何生成、生成过程如何被约束、架构判断如何进入工作流，以及为什么长期思考会直接影响最终系统的质量。
